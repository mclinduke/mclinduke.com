Diagnosis · Mobile hero overlap

The header was eating the headline.

Left: current state on a short mobile viewport (iOS Safari with URL bar visible). The fixed header sits on top of the H1. Right: the fix already shipped in css/style.css.

Root cause

  • .hero had no padding-top. With align-items: flex-end, tall content overflowed the top of the hero flex container and collided with the fixed header.
  • min-height: 100vh on iOS Safari includes the URL bar height in the viewport calc, causing the hero to clip and layout to jump when the bar shows/hides.
  • .nav-links was set to z-index: 999, but .site-header is at 1000 — so the header covered the top of the opened nav panel, blocking any link that sat beneath it.
  • No -webkit-tap-highlight-color reset — iOS renders a gray box on every tap.
  • No env(safe-area-inset-*) padding — notched iPhones lose bottom real estate.
